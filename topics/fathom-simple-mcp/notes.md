# Fathom-Simple-MCP — Research Notes

## Key Findings

### Architecture & Design

- **[HIGH]** Fathom-Simple-MCP is a lightweight, read-only, FastMCP-based Python MCP server providing GET-only access to Fathom.video meeting data (recordings, transcripts, summaries, teams, team members). — [GitHub: druellan/Fathom-Simple-MCP](https://github.com/druellan/Fathom-Simple-MCP)
- **[HIGH]** FastMCP is the dominant Python MCP framework, powering ~70% of active MCP servers. FastMCP 3.0 released January 19, 2026 achieves 1M+ daily downloads. — [GitHub: fastmcp](https://github.com/jlowin/fastmcp); [Automateed: FastMCP 3.0 Review](https://www.automateed.com/fastmcp-3-0-review)
- **[MEDIUM]** TOON (Token-Optimized Object Notation) achieves 27.7–76.4% token efficiency vs. JSON, depending on data structure. Uniform arrays see highest gains; non-uniform nested data may achieve only 30% reduction. Benchmark from LogRocket; no empirical measurement specific to Fathom-Simple-MCP's actual API responses. — [LogRocket: TOON Token Efficiency](https://blog.logrocket.com/reduce-tokens-with-toon/)

### API Limits & Rate Limiting

- **[HIGH]** Fathom.video API enforces 60 requests-per-minute user-level rate limit (not per-key; all keys from a single user share the same quota). Burst limit: 10 RPS. No paid tier offers higher limits as of April 2026. — [Fathom API Help](https://help.fathom.video/en/articles/8368641); [Fathom API Limits](https://api-docs.fathom.global/limits.html)
- **[HIGH]** Open-source Fathom MCP servers (Fathom-Simple-MCP, Dot-Fun/fathom-mcp, matthewbergvinson/fathom-mcp) lack visible rate-limit handling, request queuing, or exponential backoff logic. No request queuing or exponential backoff—concurrent agent calls exhaust quota silently. Failure mode: 429 errors without retry cause missing/stale meeting data without user notification. This makes them unsuitable for production where concurrent LLM tool calls exhaust the 60-RPM quota in seconds. — [Truto: Best MCP Server for Fathom](https://truto.one/blog/best-mcp-server-for-fathom-in-2026); [Stainless: MCP Server API Key Management Best Practices](https://www.stainless.com/mcp/mcp-server-api-key-management-best-practices)
- **[MEDIUM]** Fathom API lacks server-side search endpoint; Fathom-Simple-MCP implements client-side filtering only. At scale (100+ meetings), this forces the client to fetch entire meeting lists, then filter locally, causing rapid API call overhead and rate-limit exhaustion. Server-side filtering is industry best practice. — [Truto Blog](https://truto.one/blog/best-mcp-server-for-fathom-in-2026); [Moesif: REST API Design Filtering & Pagination](https://www.moesif.com/blog/technical/api-design/REST-API-Design-Filtering-Sorting-and-Pagination/)

### Security & Credential Management

- **[HIGH]** Fathom-Simple-MCP loads API keys via environment variables (not hardcoded), but this violates OWASP MCP01:2025 standards. Static, shared secrets provide no rotation, no audit trail, no revocation, and are vulnerable to process inspection on shared systems. OWASP MCP01:2025 violation explicitly identified. Process inspection attack vector: Linux `/proc/{pid}/environ`, Windows `Get-ChildItem Env:`. Prompt injection exfiltration risk: malicious instructions can request env var dumps. Concrete mitigation: macOS Keychain (`security add-generic-password`) or reference to secrets manager. Industry best practice requires OAuth, secure vaults (HashiCorp Vault, AWS Secrets Manager), or short-lived tokens. — [OWASP MCP Top 10: MCP01:2025 Token Mismanagement](https://owasp.org/www-project-mcp-top-10/2025/MCP01-2025-Token-Mismanagement-and-Secret-Exposure); [Will Velida: Preventing MCP01 Token Mismanagement](https://www.willvelida.com/posts/preventing-mcp01-token-mismanagement-secret-exposure)
- **[HIGH]** Bus-factor risk: Fathom-Simple-MCP has single maintainer (druellan) with minimal adoption signal. GitHub metrics: 2 stars, 1 fork, 23 total commits, 1 open issue, 1 open PR. No evidence of corporate backing or funded maintenance. Community repos significantly lack security maintenance resources compared to corporate-backed servers. — [GitHub: druellan/Fathom-Simple-MCP](https://github.com/druellan/Fathom-Simple-MCP); [Phase Transitions AI: 53 Verified MCP Servers Data-Driven Guide](https://phasetransitionsai.substack.com/p/53-verified-mcp-servers-a-data-driven-guide-by-category)
- **[MEDIUM]** FastMCP dependency has 3 CVEs in 2025–2026: CVE-2025-53366 (DoS via malformed input), CVE-2025-69196 (OAuth token scope bypass), CVE-2025-62801 (Windows command injection). Patch status unknown for Fathom-Simple-MCP's pinned version. Requires FastMCP version 2.14.2 or later to mitigate. — [FastMCP Security Advisories](https://github.com/jlowin/fastmcp/security/advisories)
- **[MEDIUM]** MCP ecosystem has 30+ CVEs in Jan–Feb 2026. Systematic analysis of 2,614 MCP implementations shows: 82% vulnerable to path traversal (CWE-22), 67% to code injection (CWE-94), 34% to command injection (CWE-78), 53% rely on long-lived static secrets. Python subprocess vulnerabilities enable RCE when input is unsanitized. Fathom-Simple-MCP's vulnerability status is unknown (no third-party security audit exists). — [Practical DevSecOps: MCP Security Vulnerabilities](https://www.practical-devsecops.com/mcp-security-vulnerabilities/); [Medium: I Found RCE in Production MCP Servers](https://medium.com/@teycir_44045/i-found-rce-in-production-mcp-servers-so-i-built-a-scanner-fd17f2213cf7); [HeyUan110: MCP Security 2026](https://www.heyuan110.com/posts/ai/2026-03-10-mcp-security-2026/)

### API Capabilities & Limitations

- **[MEDIUM]** Fathom API is read-only; no write operations exposed. Users cannot create, edit, or delete meetings via MCP. Transcripts only available post-meeting after processing (no real-time access). This restricts workflows to data retrieval, reporting, and analysis. — [Truto: Best MCP Server for Fathom](https://truto.one/blog/best-mcp-server-for-fathom-in-2026)

## Counterarguments & Risks

- **Production multi-user deployment:** Fathom-Simple-MCP is marketed as simple and lightweight, but the synthesis emphasizes it is unsuitable for production multi-user environments. This is not a weakness of the code—it is a design choice (prioritize simplicity over compliance, audit trails, and rate-limit handling).
- **OWASP vs. practicality trade-off:** Environment variables are a common Python pattern for local development. OWASP MCP01:2025 standards are enterprise-focused; teams running Fathom-Simple-MCP locally may accept this risk trade-off knowingly. The risk is context-dependent (shared systems more vulnerable than isolated containers).
- **Rate-limit exhaustion may be acceptable for some workflows:** If a solo developer is making <60 queries per minute, or if queries are staggered (not concurrent), the rate limit is not a practical constraint. Risk scales with concurrency.
- **Security audit gap:** Fathom-Simple-MCP has not been independently audited, but neither have most community open-source servers. The MCP ecosystem vulnerability survey (82% path traversal) is based on a 2,614-server sample; exact impact on Fathom-Simple-MCP is unknown.

## Gaps & Unknowns

- **No third-party security audit:** Fathom-Simple-MCP has not undergone formal penetration testing or security code review. Unknown whether it is vulnerable to specific MCP ecosystem CVEs (RCE, path traversal). Required for any production consideration.
- **No empirical token reduction benchmark for Fathom-Simple-MCP specifically:** LogRocket benchmark shows TOON achieves 30–60% gains in general, but impact on Fathom-Simple-MCP's specific output (non-uniform meeting arrays with variable field sets) is unmeasured.
- **Rate-limit retry behavior undocumented:** README does not specify whether the server queues requests, fails with error, or retries exponentially when 60-request/minute limit is exceeded. Critical for production reliability.
- **Truto and Composio pricing/SLA opaque:** Managed alternatives are positioned as production-ready, but pricing and SLA commitments are behind signup walls.
- **Adoption metrics unknown:** No published data on how many users/teams run Fathom-Simple-MCP in production.
- **OAuth implementation roadmap unknown:** No indication of maintainer plans to add OAuth (OWASP MCP01:2025 recommendation).
- **Fathom webhook API support unclear:** Unknown whether Fathom API supports webhooks (would enable push-based updates vs. pull-based polling).
